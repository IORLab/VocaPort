# VocaPort 总路线图

> English version: [2026-06-26-vocaport-roadmap.en.md](/Users/jay/Code/VocaPort/docs/superpowers/specs/2026-06-26-vocaport-roadmap.en.md)

> 状态：自 2026-06-26 起生效。路线图按能力里程碑推进，不绑定固定日历。

## 1. 当前基线

- `main` 已完成 Phase 1。
- Web 已接入 Rust/WASM runtime，Desktop 已接入 Rust native runtime。
- Android 壳层已能构建，且至少一台真机已验证可正常启动。
- `.apkg` 导入预览、确认导入、学习启动、作答、会话恢复、进度重置、快照恢复已经端到端打通。
- GitHub Releases -> GitHub Pages 下载页链路已存在，但 Android 产物仍偏工程验证口径，暂时不能视为“面向外部用户的公开 Beta 发布流程”。

## 2. 路线图原则

- 采用结果驱动而不是日期驱动。每个阶段都必须有明确退出标准。
- `Rust` 继续作为唯一业务内核；`TypeScript` 只负责 UI、平台壳层和桥接编排。
- 先把 Android 做成第一版可长期使用的本地 Beta，再扩展更重的能力。
- 在 Phase 5 之前，坚持 `offline-first` 和 `local-first`；不为了“看起来完整”提前引入账号和云同步。
- 在 Phase 4 之前，不开放运行时第三方插件执行；必须先把权限、签名、兼容性边界做稳。
- 从 Phase 3 开始只保留里程碑级描述，避免在产品和技术假设还不稳定时写出伪精确计划。

## 3. 路线图总览

| 阶段 | 定位 | 关键交付 | 退出标准 |
| --- | --- | --- | --- |
| Phase 1 | 基础骨架 | Rust 核心、`.apkg` 导入、学习闭环、快照恢复、Web/Desktop 真实 runtime、Android 壳验证 | 已完成并合入 `main` |
| Phase 2 | 第一版可用 Beta | 词库库管理、字段映射确认、移动端优先工作区、当前词库选择、Beta 发布语义 | 用户可安装 Beta，导入真实文本型 `.apkg`，选择词库，恢复或重置进度，并完成完整学习闭环 |
| Phase 3 | 学习质量增强 | 媒体渲染、反馈质量提升、基础统计、多词库操作 | 文本型和媒体型词库都能稳定学习，且状态可解释 |
| Phase 4 | 签名扩展生态 | 签名模块包、安装器、权限提示、兼容性解析、只读目录客户端 | 能安全安装、禁用、回滚受信扩展，不破坏已有数据 |
| Phase 5 | 多设备与分发扩展 | 加密备份或同步、可选账号、iOS 路线、更多正式分发 | 同一用户可在多设备恢复状态，且不牺牲离线优先体验 |

## 4. 各阶段说明

### Phase 1：基础骨架（已完成）

- 这一阶段已经交付“核心是否成立”的答案。
- 重点是把单一 Rust 业务内核、桥接契约、导入链路、学习闭环和快照恢复真正接起来。
- 它证明了架构方向可行，但还不能直接等同于“已经有了第一版面向用户的应用”。

### Phase 2：第一版可用 Beta

- 这是整个项目的第一个真正版本。
- 重点不是再做一轮底层重写，而是把现有能力变成用户可以持续使用的产品闭环。

必须补齐：

- 词库列表与“当前词库”概念
- 字段映射确认和重导入决策
- 面向手机的导航、空态、错误态和恢复态
- Beta 产物命名与发布语义

明确不做：

- iOS
- 云同步和账号系统
- 运行时第三方插件执行
- 重媒体体验和多题型扩张

### Phase 3：学习质量增强

- 这一阶段把“能用”提升到“更像学习产品”。
- 重点会转到媒体资产读取与渲染、学习反馈解释、基础统计、以及多词库操作体验。
- 之所以放在 Phase 2 之后，是因为这些能力都依赖稳定的词库模型、导入决策模型和当前词库状态。

### Phase 4：签名扩展生态

- 这一阶段才开始把“marketplace-ready”从静态边界推进到真正可安装的受信扩展模型。
- 首先交付的不会是开放式社区市场，而是受控、可签名、可回滚的模块包机制。
- 这样做符合最小风险原则，也符合插件系统常见的供应链安全约束。

### Phase 5：多设备与分发扩展

- 这一阶段才适合把同步、账号、iOS 和更完整的分发策略拉进来。
- 原因很简单：同步会放大前面所有建模错误。
- 如果在本地状态机、导入模型和扩展边界还不稳定时就上云，同步只会把错误复制到更多设备。

## 5. 依赖关系

- 先做 Phase 2，再做 Phase 3，因为媒体、统计和多词库操作都依赖稳定的词库库状态。
- 先做 Phase 3，再做 Phase 4，因为扩展系统需要稳定的宿主 UI 和能力边界，不能边定义边开放。
- 最后做 Phase 5，因为同步和跨端恢复必须建立在可靠的本地状态模型之上。

## 6. 明确不前置的事项

- 不在 Phase 2 提前做 iOS。
- 不在 Phase 2 或 Phase 3 提前做账号系统。
- 不在 Phase 2 或 Phase 3 提前开放运行时第三方插件。
- 不在 Phase 2 为了追求“看起来完整”而引入云端依赖。

## 7. 结论

- Phase 1 负责证明架构成立，已经完成。
- Phase 2 负责交付第一个真正可安装、可使用、可验证的 Beta，这是当前最应该投入的主线。
- Phase 3 之后保持里程碑级推进，避免在未知数还很多的时候把计划写得假装很精确。

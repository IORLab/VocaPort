# VocaPort 下载页改版设计文档

> English version: [2026-06-27-downloads-page-refresh-design.en.md](/Users/jay/Code/VocaPort/docs/superpowers/specs/2026-06-27-downloads-page-refresh-design.en.md)

**状态：** 设计已批准，待进入实现计划

**范围：** `apps/downloads` 下载页的信息架构、视觉风格、文案组织、主题切换与前端交互

**不包含：** GitHub Releases 数据结构重构、Rust crate 变更、桥接协议调整、下载统计服务化

## 1. 背景

当前下载页已经具备基础功能：可以从 `releases.json` 渲染最新稳定版、最新预发布版、历史版本，以及内联展开发行说明。

现状问题：

- 默认强制深色，缺少与参考站一致的浅色主视觉。
- Hero 与内容区都是统一黑色大卡，层级偏平，下载入口不够聚焦。
- 空态、加载态、错误态只是简单状态卡，没有复用正式页面的结构语言。
- 顶部缺少显式暗夜模式切换按钮。
- 历史版本与重点版本的视觉权重接近，用户不容易一眼找到“现在该下载哪一个”。

## 2. 目标

本次改版只解决一个核心问题：让用户在进入页面后的几秒内，快速理解当前推荐下载项、预发布入口和历史回退入口。

具体目标：

1. 参考 LunaTV 下载页的清晰分区与顶部操作区，提升可读性和首屏判断效率。
2. 保留 VocaPort 自身语义，不直接复刻 LunaTV，而是使用更贴近“manifest / 安装包总览”的表达。
3. 将最新稳定版设为首要下载入口，最新预发布版设为次级重点入口，历史版本降级但仍可直达。
4. 增加显式暗夜模式切换按钮。
5. 首次访问默认跟随系统主题，用户手动切换后记忆偏好。
6. 让空态、加载态、错误态与正式内容态共享同一套页面骨架。

## 3. 设计约束

- 保持现有 `release-first` 数据模型，不改成 `OS-first` 信息架构。
- 继续使用 `selectReleaseSections()` 产生 `latestStable`、`latestPrerelease`、`previousReleases`。
- 保持双语切换能力。
- 不新增搜索、筛选、分页、自动轮播等非必要交互。
- TypeScript 逻辑只留在 UI 层，不引入额外后端或桥接变更。

## 4. 页面定位

下载页不是营销首页，也不是发布后台。

它的单一职责是：把 GitHub Releases 里的公开安装包，用更清楚、更可信、更容易决策的方式呈现出来。

页面面向两类用户：

- 想要直接安装稳定版的普通用户
- 想尝试新构建或回退旧版本的测试用户

## 5. 信息架构

### 5.1 顶部 Hero

Hero 采用“单一大标题 + 右上操作区”的结构。

左侧内容：

- 小号 eyebrow：`VocaPort Download Desk` / 对应中文版本
- 大标题：强调“官方安装包总览”
- 简短描述：说明稳定版、预发布版、历史版本都能在一页内直达

右侧操作区：

- `Open GitHub Releases`
- 语言切换按钮
- 暗夜模式切换按钮

### 5.2 重点下载区

重点下载区拆成两个独立区块：

1. `01 Stable`
2. `02 Preview`

每个区块都包含：

- 分区编号
- 分区标题
- 一句用途说明
- 版本名称
- 标签名与发布时间
- 安装包列表
- 发行说明展开入口
- 跳转 GitHub Release 详情入口

其中：

- `Stable` 强调“推荐安装”
- `Preview` 强调“测试 / early access”

### 5.3 Archive 历史版本区

历史版本保留为单独区块，但视觉层级低于前两个重点区。

呈现策略：

- 使用更紧凑的卡片或列表，而不是与重点版本完全同级的大卡
- 保留版本名、标签、发布时间、安装包链接
- 仍允许展开发行说明

### 5.4 Footer

底部继续显示：

- GitHub 仓库来源
- 数据更新时间

文案更接近“Last synced / 最后同步”，而不是纯技术口吻。

## 6. 视觉方向

### 6.1 总体气质

主视觉参考 LunaTV 的以下特征：

- 浅色纸面背景
- 大圆角容器
- 明确的分区边界
- 顶部集中操作区
- 低噪音、高可读性

但不直接复制其品牌语气。VocaPort 版本采用更偏“安装清单 / manifest”的表达。

### 6.2 浅色主题

浅色主题应是页面默认外观，但默认值不是硬编码浅色，而是首次访问跟随系统设置。

视觉要点：

- 背景使用偏暖灰或雾白色，不是纯白
- 卡片像纸面或仪表板桌面，而不是玻璃拟态面板
- 使用轻网格或细分隔线增强结构感
- 主文字使用深蓝灰或墨色，提高可读性

### 6.3 暗夜主题

暗夜主题不是简单反色，而是保留同样的分区结构和材料感。

视觉要点：

- 背景改为深墨蓝或深炭色
- 卡片仍有边界和层次，但减少过亮高光
- 编号、标签和主操作保持足够对比度

### 6.4 排版

- Hero 标题采用更有辨识度的 display / serif 风格
- 正文、元信息和按钮采用更克制的 sans-serif
- 分区编号、eyebrow、标签统一用更窄、更结构化的样式

### 6.5 结构元素

可保留的结构性元素：

- 编号 `01 / 02`
- eyebrow
- 分隔线
- 状态标签

应移除或弱化的旧风格：

- 整页统一的深色玻璃氛围
- 所有卡片权重过于接近的问题

## 7. 文案策略

### 7.1 Hero 文案

Hero 文案从“All builds, one page”升级为更明确的安装页定位，例如：

- 中文：`官方安装包总览`
- 英文：`Official build manifest`

副文案需要表达三件事：

- 最新稳定版可以直接安装
- 预发布版适合试用与验证
- 历史版本可用于回退

### 7.2 分区说明

- `Stable`：强调“推荐给大多数用户”
- `Preview`：强调“适合测试新功能或提前验证”
- `Archive`：强调“回退或查找旧版本”

### 7.3 状态文案

加载态、错误态、空态都要与主页面语气一致：

- 不使用生硬的技术提示
- 明确告诉用户当前发生了什么
- 在错误态中给出重新查看 GitHub Releases 的路径

## 8. 交互设计

### 8.1 语言切换

- 保留现有中英文切换
- 语言切换继续影响 Hero、分区标题、状态文案和发行说明本地化

### 8.2 暗夜模式切换

主题切换规则：

1. 首次访问时，读取 `prefers-color-scheme`
2. 若用户已手动选择主题，则优先使用保存值
3. 用户点击切换按钮后，立即切换并写入 `localStorage`

实现策略：

- 使用 `data-theme="light" | "dark"` 驱动 CSS 变量
- 不额外暴露 “auto” 第三种 UI 状态
- 按钮只负责在亮/暗之间切换

### 8.3 发行说明展开

- 继续使用当前 inline expand 方式
- 在新版视觉中改为嵌入式说明面板
- 保持双语发行说明解析逻辑

### 8.4 外链行为

以下链接继续新窗口打开：

- GitHub Release 页面
- 安装包下载链接

## 9. 状态设计

### 9.1 Loading

加载态使用与正式页面相同的 Hero 外壳与主题系统，只在内容区显示“正在同步安装包列表”的状态面板。

### 9.2 Error

错误态使用统一骨架，并在状态面板中明确：

- release 数据暂时无法加载
- 用户仍可去 GitHub Releases 查看公开版本

### 9.3 Empty

空态不应看起来像异常，而应像“暂时还没有公开构建”。

页面仍显示 Hero 与顶部操作区，并在主体区显示：

- 暂无公开安装包
- 一旦 GitHub Release 附加资产，这里会自动更新

### 9.4 Filled

Filled 状态使用完整三段结构：

- Stable
- Preview
- Archive

## 10. 组件边界

本次改版建议在现有文件基础上，抽出更清晰的展示单元，但不做无关重构。

建议边界：

- `DownloadsPage`
  - 持有 `locale` 与 `theme`
  - 负责根据目录数据决定渲染哪些区块
- `Hero`
  - 渲染标题、副文案、顶部操作区
- `ThemeToggle`
  - 只处理主题切换按钮表现
- `ReleaseSpotlightCard`
  - 用于 `Stable` 与 `Preview`
- `ArchiveReleaseList`
  - 用于历史版本区
- `StatePanel`
  - 统一 Loading / Error / Empty 的面板骨架

## 11. 响应式与可访问性

### 11.1 响应式

- 桌面端 Hero 使用左右布局
- 窄屏下顶部操作区堆叠到标题下方
- `Stable` 与 `Preview` 在桌面端可以并排，在移动端必须单列
- Archive 区在移动端保持紧凑但可点按

### 11.2 可访问性

- 所有主题切换与语言切换按钮都要有明确 `aria-label`
- 亮暗主题的对比度必须满足基础可读性要求
- 键盘可访问路径不能依赖 hover
- 展开/收起发行说明继续使用 `aria-expanded`

## 12. 测试要求

本次改版至少覆盖以下前端测试：

1. 渲染新版 Hero 与三段结构
2. 语言切换后标题与发行说明同步变化
3. 暗夜模式切换按钮可以切换主题
4. 已保存主题偏好会在初始化时生效
5. 空态仍沿用新版页面骨架
6. 发行说明展开逻辑在新版布局下保持可用

## 13. 非目标

本次不做以下事情：

- 不改 GitHub Releases 抓取逻辑
- 不新增按操作系统筛选
- 不新增下载量排序
- 不把下载页改造成完整营销主页
- 不引入设计系统级依赖或大型 UI 库

## 14. 实现入口

预计主要改动集中在：

- `apps/downloads/src/App.tsx`
- `apps/downloads/src/index.css`
- `apps/downloads/src/i18n.ts`
- `apps/downloads/src/App.test.tsx`

如果实现中需要新增小型展示组件，可以继续留在 `apps/downloads/src` 内部，不扩散到共享包。

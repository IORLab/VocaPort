# VocaPort UI 与交互设计文档

> English version: [2026-06-26-vocaport-ui-interaction-design.en.md](/Users/jay/Code/VocaPort/docs/superpowers/specs/2026-06-26-vocaport-ui-interaction-design.en.md)

**状态：** 已确认体验方向，暂不进入实现设计

**范围：** 第一阶段 UI、视觉风格、信息架构、交互流程

**暂不包含：** Rust/WASM/Tauri 架构、存储设计、模块契约、代码组织、实现细节

## 1. 产品定位

**产品名：** `VocaPort`

**副标题：** 连接词汇、练习与进步

**定位描述：** 一个以词库导入为起点、以日常练习为核心、同时支持轻松养成和高强度学习的词汇学习应用。

## 2. 体验目标

- 让首次导入足够顺利，用户能快速获得“导入成功”的确定感。
- 让日常学习足够轻松，愿意每天打开。
- 让高强度学习有专门模式，但不把产品割裂成两套系统。
- 让产品首先看起来像学习应用，而不是后台管理系统，也不是营销官网。
- 为后续扩展能力预留位置，但第一阶段不让“扩展”抢走学习主线。

## 3. 核心设计原则

- **容易开始，容易回到主线。** 用户随时都应该知道下一步做什么。
- **一屏一个主任务。** 每个页面都必须有明确主动作。
- **轻游戏化，但不幼稚。** 要有成长感和轻反馈，但不能做成噪音型教育产品。
- **专注模式是壳层切换，不是第二个产品。** 只改变表现和节奏，不改变学习数据语义。
- **进步必须可感知。** 用户应持续看到掌握感、完成感和积累感。
- **配置是次要路径。** 设置、导入细项、高级选项不能打断主学习路径。

## 4. 视觉方向

VocaPort 应该呈现为一款现代学习应用：有轻度成长感，但整体安静、克制、稳定。

### 4.1 品牌关键词

- 安静
- 模块化
- 鼓励感
- 有结构
- 稳定成长

### 4.2 视觉气质

- 大圆角和充足留白
- 浅雾色背景，而不是纯白硬底
- 强调色集中用于进度和主动作
- 插图使用极少
- 图标以卡片、节点、连接、进度为核心语义

### 4.3 明确避免的风格

- 过度儿童化、卡通化的教育产品风格
- 强刺激、强庆祝、强弹窗的游戏化风格
- 冷冰冰的数据后台风格
- 类似内容消费型首页的布局节奏

## 5. 学习模式设计

第一阶段采用一套学习引擎、两种表现模式。

### 5.1 成长模式

**默认模式**

用于日常学习、轻松打卡和习惯养成。

**特征**

- 明确展示连续学习天数
- 有轻量目标感和完成感
- 正确反馈更柔和
- 动效略多一点
- 节奏更亲和，适合短时学习

### 5.2 专注模式

用于高强度学习、长时间刷卡和低干扰练习。

**特征**

- 动效明显减少
- 视觉刺激降低
- 信息密度更高
- 切题更快
- 文案更直接

### 5.3 模式切换会影响什么

- 颜色强度
- 动效强度
- 页面密度
- 文案语气
- 答题节奏
- 首页重点信息排序

### 5.4 模式切换不会影响什么

- 词库数据
- 复习记录
- 进度语义
- 题目内容
- 调度结果
- 学习历史

## 6. 信息架构

第一阶段采用学习主线优先的信息架构。

### 6.1 一级区域

1. `Home`
2. `Review`
3. `Library`
4. `Progress`
5. `More`

### 6.2 结构说明

- `Home` 是每天重新进入应用时的起点。
- `Review` 是最核心的学习场景。
- `Library` 负责词库导入和词库管理。
- `Progress` 负责呈现掌握趋势和学习负载。
- `More` 负责设置、模式控制和未来扩展入口。

`More` 必须存在，但在第一阶段不能与学习主线竞争注意力。

## 7. 导航设计

### 7.1 移动端 / 安卓

底部四栏导航：

- `Home`
- `Review`
- `Library`
- `Progress`

`Review` 需要成为视觉上的主动作。

`More` 不放到底栏，放在右上角菜单或头像菜单中。

### 7.2 桌面端 / Web

左侧导航栏：

- `Home`
- `Review`
- `Library`
- `Progress`
- `More`

右上角工具位：

- 模式切换
- 搜索
- 设置 / 用户菜单

## 8. 页面设计

## 8.1 Home

**页面角色：** 每日学习入口

**核心目标：** 用户进入后 3 秒内理解“今天该做什么”

### 页面骨架

1. 欢迎与连续学习区
2. 主学习入口
3. 今日任务摘要
4. 最近词库
5. 本周成长反馈
6. 次级操作入口

### 关键组件

- 连续学习天数
- 今日目标进度
- `继续学习` 按钮
- 待复习数量
- 新词数量
- 预计时长
- 最近学习词库
- 本周完成量摘要
- `恢复当前会话` 入口（如存在）

### 首次空状态

适用于还没有导入任何词库的用户。

- 标题：`从你的第一个词库开始`
- 描述：`导入 Anki 词库，开始你的第一轮练习。`
- 主按钮：`导入 Anki 词库`
- 次按钮：`查看示例流程`

### 两种模式下的差异

**成长模式**

- 更强调 streak 和目标环
- 色彩更温和但更有鼓励感
- 成长反馈更靠前

**专注模式**

- 更强调待学习数量和预计时间
- 装饰性反馈减少
- 快速开始更靠前

## 8.2 Review

**页面角色：** 主练习场景

**核心目标：** 让用户尽可能低摩擦地连续答题

### 页面骨架

1. 会话进度区
2. 题干与媒体区
3. 四选项区
4. 正误反馈区
5. 底部动作区

### 关键组件

- 进度条
- 剩余题数
- 剩余预计时间
- 单词或题干
- 图片 / 音频媒体区域
- 四个混合选项
- 正误反馈
- `下一题`
- `标记太难`
- `查看详情`

### 设计规则

- 一次只展示一张卡
- 页面始终只有一个明显主动作
- 选项在移动端和桌面端都必须易扫读
- 反馈不能长时间打断下一题
- 如果是恢复会话，题干、选项顺序和剩余题数必须与中断前一致

### 两种模式下的差异

**成长模式**

- 轻量成功动效
- 留白更大
- 反馈更柔和
- 会话节奏更轻松

**专注模式**

- 更少过场
- 更高密度
- 更克制的颜色
- 更直接的反馈语气

## 8.3 Library

**页面角色：** 词库管理与导入中心

**核心目标：** 让导入和管理看起来像引导式工具，而不是后台配置台

### 页面骨架

1. 顶部操作区
2. 词库列表
3. 词库详情
4. 导入向导入口

### 词库列表信息

- 词库名称
- 单词数量
- 最近学习时间
- 掌握进度

### 词库详情信息

- 总览统计
- 导入来源摘要
- 字段映射摘要
- 历史导入摘要
- 重复导入 / 更新摘要
- 重置学习进度入口

## 8.4 Import 向导

导入流程必须是引导式向导，不是表单密集型后台流程。

### 固定四步

1. `选择文件`
2. `解析预览`
3. `确认字段`
4. `导入结果`

### 每一步说明

**选择文件**

- 支持拖拽和文件选择
- 明确显示支持格式：`.apkg`

**解析预览**

- 显示识别到的词条数
- 显示媒体数量
- 显示导入到的复习历史数量
- 提示是否检测到复杂字段结构

**确认字段**

- 单词字段
- 释义字段
- 例句字段
- 图片字段
- 音频字段

自动识别足够准确时，默认给出推荐映射。

**导入结果**

- 成功导入数量
- 跳过数量
- 媒体导入摘要
- 历史导入摘要
- 重复文件或词库更新摘要
- 明确下一步入口

### 导入设计要求

- 每一步都要告诉用户系统正在做什么
- 错误必须可恢复
- 警告必须清晰，但不制造恐慌
- 必须明确区分“完全重复文件”和“同词库新版本导入”
- 成功导入后必须给出强主动作 `开始学习`

## 8.5 Progress

**页面角色：** 进步反馈与学习负载查看

**核心目标：** 让用户看到成长，而不是被报表压住

### 页面骨架

1. 总览卡片
2. 趋势区域
3. 负载区域
4. 历史与重置区域

### 总览内容

- 连续学习
- 已掌握词数
- 本周完成量

### 趋势内容

- 复习量趋势
- 掌握增长趋势

### 负载内容

- 待复习分布
- 即将到期的学习压力

### 历史内容

- 最近重置记录
- 导入历史摘要

### 两种模式下的差异

**成长模式**

- 更强调长期成长和连续性

**专注模式**

- 更强调效率和当前负载

## 8.6 More

**页面角色：** 次级控制区与未来扩展入口

第一阶段应存在，但要弱化。

### 第一阶段内容

- 学习模式切换
- 动效强度控制
- 主题偏好
- 导入器信息
- 调度器信息
- 数据管理
- 关于

### 后续可扩展方向

- 模块列表
- 扩展详情
- 市场入口
- 安装 / 启用 / 禁用视图
- 权限说明

第一阶段不直接把这里命名为 `Marketplace`。

## 9. 核心用户流程

## 9.1 首次用户主流程

1. 用户进入应用
2. 在 `Home` 看到空状态
3. 点击导入 Anki 词库
4. 选择 `.apkg`
5. 查看解析预览
6. 确认字段映射
7. 完成导入
8. 查看导入结果
9. 开始第一次学习

### 流程目标

让用户从空状态尽快进入第一张题卡。

## 9.2 日常回访流程

1. 用户打开应用
2. 在 `Home` 看到今日待学摘要
3. 如存在未完成会话，优先提示 `恢复当前会话`
4. 点击 `继续学习`
5. 完成一轮短会话
6. 回到成长反馈

### 流程目标

尽量降低每天重新开始学习的门槛。

## 9.3 高强度学习流程

1. 用户进入 `Home`
2. 切换到专注模式
3. 进入 `Review`
4. 连续学习
5. 以简洁效率摘要结束本轮

### 流程目标

让用户感觉自己进入了“训练状态”，但仍在同一个产品中。

## 9.4 异常与恢复流程

第一阶段必须显式考虑以下中断点：

- `.apkg` 解析失败
- 字段识别失败
- 媒体缺失
- 历史导入无法完全映射
- 学习中途退出
- 用户请求重置学习进度

### 恢复策略

- 用自然语言说明失败原因
- 自动识别失败时进入手动映射
- 媒体缺失时回退为文字 / 例句选项
- 历史无法完全映射时显示摘要和警告
- 中断后允许恢复当前会话，且恢复后的题干、选项和剩余题数必须保持一致
- 完全重复文件与同词库更新要给出不同提示
- 重置只重置当前学习状态，不抹掉原始导入历史

## 10. 微交互方向

### 10.1 正确反馈

- 短促、明确
- 不用长时间庆祝打断学习
- 反馈后下一步应立即明确

### 10.2 错误反馈

- 信息清楚
- 不带羞辱感
- 允许快速恢复到下一题

### 10.3 模式切换

- 必须即时、可逆、可理解
- 切换后要让用户感知到节奏和视觉变化
- 不要求退出当前学习流

### 10.4 加载态

- 导入解析阶段尤其重要
- 必须显示进度和通俗状态说明

### 10.5 空状态

- 任何空状态都应引导下一步
- 不能出现“空白但不知道怎么办”的死角

## 11. 文案语气

### 11.1 成长模式

- 温和
- 鼓励
- 简短
- 低压力

### 11.2 专注模式

- 安静
- 直接
- 高效
- 少打扰

## 12. 可用性与无障碍基线

- 两种模式下都必须保持清晰文本对比
- 移动端点击区域必须足够大
- 动效必须可降低
- 桌面端 / Web 必须支持良好的键盘导航
- 进度信息和选项信息必须支持快速扫读
- 当图片选项和文字选项混合出现时，布局仍需稳定易读

## 13. 第一阶段明确不做的 UI

- 不做完整插件市场界面
- 不做复杂社交功能
- 不做重度游戏化经济系统
- 不做报表中心式复杂数据看板
- 不做按平台完全分裂的多套视觉系统

## 14. 总结

第一阶段的 VocaPort 应该首先是一款学习应用，其次才是一个可扩展的平台。默认体验应以轻游戏成长型为主，让用户愿意每天回来；专注模式则承接高强度训练需求。导入、学习、进度三者必须形成一条连续、自然、低摩擦的学习路径。
